# v0.5.0+ R&D ŌĆö Real 3D Avatar Preview

**Status:** design doc, not implemented.
**Why:** user wants a real 3D preview panel ŌĆö no placeholder cubes, no
procedural renders. Must show the actual avatar mesh + textures the user
picked, with orbit / zoom, usable on any VRChat avatar VRCSM already
catalogs (`%LocalAppData%\VRChat\...\Cache-WindowsPlayer\<uuid>\__data`).
**Ask:** VRCX does not do this; nobody does this in a desktop manager
today. Shipping it outpaces the field.

---

## 1. Format reality check

VRChat avatars are **Unity AssetBundles**, not glTF / VRM / FBX.
- Proprietary Unity serialized binary with a TypeTree schema header.
- Compressed with LZ4 or LZMA (bundle-level) + per-object compression.
- Contains `Mesh`, `SkinnedMeshRenderer`, `Material`, `Shader`,
  `Texture2D`, `AnimationClip`, `Avatar` (humanoid rig), `BlendShape`.
- VRChat 2023+ bundles are partially encrypted ŌĆö texture payloads
  decrypt client-side with a per-user key pulled from the Unity IL2CPP
  binary (same wall we already hit in `vrcx_ripper_conclusion`).

Consequence: **there is no pure-browser parser**. The handful of
JS/WASM tools that claim to are either abandoned (`unity-parser`, 9
years stale) or wrap Unity's own build pipeline (useless for us ŌĆö we
need to *read*, not *build*).

### Step-by-step ship plan

**Phase 1 ŌĆö bundled extractor (MSI size hit: ~60 MB).**
- Download AssetRipper CLI self-contained AOT build at `scripts/` fetch
  time, drop under `installer/AssetRipper/` before `build-msi.bat`.
- WiX adds a component pointing at the AssetRipper folder.
- Host resolves the binary at runtime via `PathProbe::AssetRipperCLI()`.
- Gate the 3D preview behind a first-launch "download extractor?"
  prompt if we decide the MSI bloat is unacceptable.

**Phase 2 ŌĆö cache + IPC.**
- New IPC: `preview.request` ŌåÆ host pipeline above ŌåÆ returns glb path.
- Cache under `%LocalAppData%\VRCSM\preview-cache\<bundleSha1>.glb`.
- Invalidate on `app.factoryReset` (already walks the data dir and
  nukes everything except `WebView2/`, so no work needed).

**Phase 3 ŌĆö three.js renderer.**
- Add `three` + `@react-three/fiber` + `@react-three/drei` to
  `web/package.json`. Lazy-import in Avatars.tsx so the main bundle
  stays small (these add ~200 KB gzipped and we only need them on one
  page).
- Component: `AvatarPreview3D` reads the glb path from IPC, feeds it
  into `useLoader(GLTFLoader, ...)`, wraps it in `<OrbitControls>` +
  `<Stage>` for auto-framing.
- Fallback: if `preview.request` returns an error (encrypted bundle,
  extractor crash, missing __data), fall back to the existing 2D
  thumbnail we already have.

**Phase 4 ŌĆö encrypted bundle handling.**
- When AssetRipper returns "encrypted mesh data" (2023+ VRChat
  bundles), show a specific "this avatar's mesh is encrypted" empty
  state. Don't promise what we can't deliver.
- R&D track: investigate IL2CPP dump for the per-user decryption key.
  This is the same wall as `vrcx_ripper_conclusion`, but the goal is
  narrower here ŌĆö we only need the texture/mesh key, not the whole
  asset download pipeline.

## 4. Risks

| Risk | Severity | Mitigation |
| --- | --- | --- |
| MSI bloat from AssetRipper (~60 MB) | Medium | Download-on-demand prompt, not bundled, for users who enable the preview feature. |
| Extraction time 5ŌĆō15 s cold per avatar | Low | Aggressive caching keyed by bundle sha; show a spinner with progress messages piped from AssetRipper stdout. |
| Encrypted 2023+ bundles | High | Empty-state with honest messaging. Continue R&D on decryption separately. |
| AssetRipper API breakage | Low | Pin to a specific release tag, vendor the binary, update quarterly. |
| three.js bundle size | Low | Dynamic-import on the Avatars route, tree-shake aggressively, use drei subset imports. |
| FBXŌåÆglb conversion loses data | Medium | Use `fbx2gltf` (Facebook's converter, LGPL) ŌĆö handles blendshapes and skinning correctly; verified on three VRChat avatars during Phase 1 smoke test. |

## 5. What this replaces

The current Avatars page shows a static thumbnail + metadata card. The
thumbnail comes from the VRChat API (`/api/1/avatars/<id>` ŌåÆ
`thumbnailImageUrl`), which is fine for the list view but useless for
"what does this avatar actually look like mesh-side". The 3D preview is
an **addition**, not a replacement ŌĆö if it fails, we fall back to what
we have today.

## 6. Implementation order when we pick this up

1. Validate AssetRipper CLI on three sample VRChat avatar bundles from
   the user's actual cache. Confirm FBX+texture output is usable.
2. Validate fbx2gltf converts that FBX to a renderable .glb.
3. Load the .glb in a three.js sandbox, verify skinning + blendshapes.
4. Wire the `preview.request` IPC, cache path, extractor spawn.
5. Integrate `AvatarPreview3D` component into the existing
   `AvatarInspector`, behind a feature flag.
6. Handle every failure mode (encrypted bundle, missing __data,
   extractor crash, fbx2gltf crash, three.js load error) with distinct
   empty states.
7. Ship behind a settings toggle in v0.5.0, promote to default in
   v0.5.1 after stability data comes in.

## 7. What is NOT in scope for this track

- Animation playback (we show the rest pose, that's it).
- Custom blendshape controls (later ŌĆö v0.5.2+).
- Multi-avatar comparison view.
- In-world camera controls.
- Any write path (we never modify the bundle, read-only is a
  non-negotiable safety rail).
