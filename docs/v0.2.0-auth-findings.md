## 1. 注册表:VRChat 自己的持久化层

### 1.1 位置

```
HKEY_CURRENT_USER\Software\VRChat\VRChat
```

7 条相关 `REG_BINARY` value(都是 ASCII 字节 + null 结尾,内容是 base64 字符串):

| Value 名 | 解出的 PlayerPrefs key | 密文长度 | 含义 |
|---|---|---|---|
| `14C4B06B824EC593239362517F538B29_h1518735675` | `username` | 16B | 用户名 |
| `5F4DCC3B5AA765D61D8327DEB882CF99_h997075211` | `password` | 16B | 密码 |
| `BD2E932A03A19217AB5A1DFB5AA93340_h1920873149` | `authToken` | 57B | auth cookie |
| `785C2BDD2C43070A10BC35E5E687A467_h2366186322` | `twoFactorAuthToken` | 525B | 2FA cookie |
| `93D3AE97F80BEDA8E396065DC4770A93_h1522417508` | `humanName` | 16B | 显示名 |
| `E1F946CE2FD302B954E26AD92C0B30BF_h577412776` | (未知)| 16B | (未对上) |
| `BCD9D91ED8D8F1926B20D3D620647C8E_h2167456022` | (未知)| 16B | (未对上) |

### 1.2 Value 名生成公式(已实证)

```
registryValueName = MD5(UTF8(field)).hex_upper + "_h" + djb2a(MD5_hex_upper)
```

- `MD5` 输出**大写 hex**(注册表里看到的就是大写,不是小写)
- `_h<num>` 后缀是 Unity PlayerPrefs 的 **djb2a**(不是 SHA-1,不是 FNV):
  ```python
  def djb2a(s):
      h = 5381
      for c in s:
          h = ((h * 33) ^ ord(c)) & 0xFFFFFFFF
      return h
  ```
- 5/5 已知字段反算成功(`username`/`password`/`authToken`/`twoFactorAuthToken`/`humanName`),公式确认。

### 1.3 加密 — 算法 vs key

#### 算法部分(高置信)

VRChat 用的是 `rawandnf/SecurePlayerPrefs` 的 fork,源码看得到:

```csharp
// DESEncryption.Decrypt
var des = new DESCryptoServiceProvider {
    Mode = CipherMode.ECB,
    Padding = PaddingMode.PKCS7,
    Key = Encoding.ASCII.GetBytes(privateKey),  // 8 ASCII bytes
};
using var ms = new MemoryStream(Convert.FromBase64String(encryptedString));
// ...
```

所以解密一条 value 的标准流程是:

```
ASCII bytes → trim trailing NUL → base64 decode → DES-ECB-PKCS7 decrypt with 8B ASCII key
```

#### Key 部分(无解)

rawandnf 原版:

```csharp
private static string privateKey = "abcd" + UnityEngine.Random.Range(100, 1000) + "h";
// 1000 个候选,用 "abcdefgh" 加密后存到 PlayerPrefs.RAND_KEY
```

我们的现实:

- ❌ 暴力 `"abcd" + 100..999 + "h"`(900 候选)→ **0 命中**(PKCS7 unpad 全错)
- ❌ 扩到 `"abcd" + 1..99999 + "h"`(10w 候选)→ 0 命中
- ❌ `"abcd" + (3 letters) + "h"`(17576 候选)→ 0 命中
- ❌ 把 `metadata_strings.json` 里所有 5082 条 8 字符 ASCII 可打印串当作 key → 0 命中
- ❌ 注册表里 **没有** `MD5("abcdefgh")_h245213793` 这条 → 说明 VRChat 把 RAND_KEY 也藏起来了
- ❌ `metadata_strings.json` 里 **没有** `abcd`/`efgh`/`abcdefgh`/`RAND_KEY`/`PRIVATE_KEY` 任何一个

**结论**:VRChat 自己 fork 了 SecurePlayerPrefs,key 生成方式被改了,真正的 key 只存在于编译后机器码里(`DESEncryption.Encrypt` 在 `0x7FFB56ACA720`)。

要破出这个 key 只剩两条路:
1. **IDA / Ghidra 反汇编** `0x7FFB56ACA720`,反推 key 派生函数
2. **Frida runtime hook** `DESEncryption.Encrypt` 抓 `Key` 参数

两条都**不在 v0.2.0 sprint 范围内**。即使破出来,也只是解到旧用户的历史明文,不解决"VRCSM 自己怎么做 auth"。

### 1.4 加密 vs 混淆 — 实证

`humanName` 解 base64 出来:

```
5cb57de92b34ca4ab9ec14d105d0dac1
```

字节分布均匀,**确认是真加密**,不是 XOR / ROT 之类的伪混淆。

## 2. VRChat 真正的"自动登录"是 Steam ticket

### 2.1 实证

`D:\WorkSpace\VRChat\VRChat_Data\il2cpp_dump_tools\output\trace_auth_capture.txt`(Frida 实际抓的):

```
[+0.0s] Steam connection initialized
[+0.2s] [Behaviour] authenticating
[+0.3s] User Authenticated: dwgx (usr_xxx) — via VRChat API
```

### 2.2 流程

```
Steam client → SteamUser.GetAuthSessionTicket()
   ↓
VRChat client → POST /api/1/auth/steam (with ticket)
   ↓
VRChat API → 返回 JWT auth cookie
   ↓
Photon Custom Auth → 房间连接
```

关键 IL2CPP RVA(`output/eac_auth_analysis.md` 已经做完):

| 函数 | RVA |
|---|---|
| `VRCFlowManager.AuthenticateWithSteam` | `0x1F1A8A0` |
| `GetKeyServerAuthToken` | `0x112A080` |
| `OnCustomAuthResponse` | `0x211A240` |

### 2.3 推论

1. 注册表里那 7 条 `SecurePlayerPrefs` value 是 **Steam 之前** 的"用户名密码登录"残留。Steam 用户即使重装也能登录,因为 Steam ticket 流程根本不读这些。
2. **VRCSM 不需要复制 Steam ticket 流程**。VRCSM 是 standalone 第三方 shell,没有 Steam SDK 也没有 EAC,直接走 VRChat 的 HTTP / WebView 登录就行。
